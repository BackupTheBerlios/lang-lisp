Unit tests for the Core development stage

I will only unit test things if the test does not constitute reimplementing it. Also the thing tested should not be too simple. Good things to test are functions of which there is another function that should be its inverse, and structures on which there are demands on what its state is like.

Type getting/setting tests

After setting a bunch of types, subsequent getting should get things correctly. Should also try getting less specific types then exist. It should work regardless if use-hash is on or off. Perhaps should also include special types, but this might be inpractical. The resulting state of the structure holding the types should forfill its state correctness. Where state correctness is the following:

  • If you go down the more-specific slot of a typeset, it should always actually be more specific.
  • What is in the more-specific list should never be coarser then each other.
  • The more-specific list should be sorted to preference. Also, if a is preferred over b, b should not be prefered over a.
